Contexto e Desafio

Problema
🏛️ Sistema Legado SOAP
Sistema legado Oracle + SOAP armazena dados de biometria facial de clientes Vivo. Não possui autenticação moderna, não emite métricas e não pode ser modificado sem alto risco.
⚠️ Problema
Parceiros de negócio precisam consultar dados biométricos via REST/JSON com autenticação OAuth2 — protocolo e formato incompatíveis com o legado SOAP.
🔒 Restrição LGPD
Biometria facial = dado sensível (Art. 11 Lei 13.709/2018). CPF = dado pessoal. Nunca podem ser logados em claro. Acesso deve ser auditado com correlationId.
✅ Solução Proposta
Plataforma de integração com 3 serviços: Gateway (borda OAuth2), Core API (REST + ACL), Legacy SOAP (adaptado). Defesa em profundidade com segredos internos.
🎯 Objetivos
• Expor SOAP legado via REST/JSON moderno
• OAuth2 Client Credentials (JWT RS256)
• Zero acesso direto ao legado
• Observabilidade completa (métricas + logs + traces)
• Estratégia de migração sem big bang
• Conformidade LGPD desde o design

Modelo de Segurança — Defesa em Profundidade

Security
1
OAuth2 / JWT RS256 — Keycloak 24
Client Credentials · Scope biometria:read · Validação stateless via JWKS · Expiração 5min · Realm vivo-poc
2
Spring Cloud Gateway — Filtro de Roteamento
GatewaySecurityConfig: /api/v1/** exige authenticated(). /management/** e /actuator/** permitAll() apenas na rede interna. anyExchange().denyAll()
3
X-Gateway-Secret — Core API (GatewaySecretFilter)
Header interno injetado pelo Gateway em cada request. @Order(HIGHEST_PRECEDENCE+1) — verifica antes de qualquer controller. 403 se ausente ou inválido. Impede bypass direto.
4
X-Core-Secret — Legacy SOAP (InternalSecretInterceptor)
EndpointInterceptor no biometria-legacy-soap. Retorna SOAP Fault UNAUTHORIZED se header ausente ou inválido. Impede acesso direto ao SOAP legado por qualquer cliente.
5
LGPD — Privacy by Design
CpfMasker: *********XX em todos os logs · imagemBase64 nunca logada · JWT/secrets nunca logados · correlationId em toda linha de log para rastreabilidade de auditoria

Estratégia de Migração — Strangler Fig Pattern

Migration
1
Fase 1 — Estabilização e Instrumentação
0–3 meses
Substituir H2 por Oracle real (HikariCP) · Circuit Breaker Resilience4j com fallback 503+Retry-After · Prometheus + Grafana em produção · Audit log LGPD imutável · TLS em todas as comunicações
2
Fase 2 — Strangler Fig: Banco Próprio + CDC
3–12 meses
Schema próprio PostgreSQL/Oracle para Core API · Debezium CDC captura mudanças do Oracle legado → Kafka · Consumer Kafka sincroniza dados locais (lag < 1s) · Feature flag: % leituras → banco local vs SOAP · Evento biometria.consultada para auditoria
3
Fase 3 — Microsserviço Autônomo
12–24 meses
Desligar biometria-legacy-soap após 30 dias sem chamadas SOAP · Migrar para OpenTelemetry (OTLP → Tempo) · CQRS se leitura > 10:1 escrita · DPIA revisada e aprovada pelo DPO · Go-live com SLA 99,9%
💡 Por que Strangler Fig e não reescrita total? Zero downtime, risco controlado fase a fase, o legado permanece operacional durante toda a transição. Feature flags permitem rollback imediato em qualquer fase sem deploy.

Como Executar a Aplicação

Getting Started
🐳 Stack Completa com Docker Compose
Sobe apps + Keycloak + observabilidade em um único comando
# 1. Clonar e entrar na pasta git clone <repo> && cd poc-vivo-integracao-arquitetural # 2. Subir stack completa docker compose -f docker-compose.yml \ -f docker-compose.observability.yml \ up -d # 3. Obter token JWT TOKEN=$(curl -s -X POST \ http://localhost:8081/realms/vivo-poc/protocol/openid-connect/token \ -d "grant_type=client_credentials&client_id=parceiro-externo&client_secret=parceiro-secret" \ | jq -r .access_token) # 4. Consultar biometria curl -s -H "Authorization: Bearer $TOKEN" \ http://localhost:8080/api/v1/biometria/1234567890 # 5. Swagger UI (autenticar com oauth2) open http://localhost:8080/swagger-ui/index.html
